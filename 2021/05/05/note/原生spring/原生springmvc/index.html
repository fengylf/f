DispatcherServlet继承关系

DispatcherServlet>FrameworkServlet>HttpServletBean>HttpServlet

  • HttpServletBean:

    • final掉HttpServlet的init(),同时留出了initServletBean()来取代;

  • FrameworkServlet:

    • 实现了ApplicationContextAware,持有了ac.
    • 重写并final了initServletBean(),同时留出了initFrameworkServlet()来取代之;此方法中initWebApplicationContext();有初始化wac的能力,但可以被子类覆盖[注解驱动就提前初始化好了wac,]
    • 重写service(),用processRequest()方法用来统一处理request请求,留出了doService(request, response)给子类重写;
    • ==ContextRefreshListener内部类来监听ac的ContextRefreshedEvent事件,并留出了onRefresh()空方法给子类==

  • DispatcherServlet:

    • ==重写了onRefresh(),来初始化九大组件.==
    • 重写了doService(),并引出doDispatch(request, response)

DispatcherServlet请求处理流程

  • DispatcherServlet是前端控制器,用于分发request到对应的handler.
  • HanlerMapping:映射请求url与controller.
  1. request到达DispatcherServlet
  2. 根据请求,遍历hanlerMappings,HandlerExecutionChain handler = mapping.getHandler(request),直到得到一个.
  3. 对应的HanlerMapping返回HandlerExecutionChain(包含处理该请求的handler),交给DispatcherServlet
  4. 根据得到的handler/chain,遍历HandlerAdapters,找到支持该handler的adapter.
  5. 通过HandlerAdapter去执行对应的handler,得到ModelAndView,交给DispatcherServlet.
  6. DispatcherServlet调用ViewResolver去处理mav,得到view对象.
  7. 渲染view,将结果返回给用户.

九大组件

  1. MultipartResolver(context);

    初始化时根据”multipartResolver”为id,MultipartResolver类型,==只能一个==.而且你自己配置的话id必须正确.

    MultipartResolver用于处理文件上传.解析request成MultipartFile,并封装到MultipartHttpServletRequest中.

    MultipartResolver两个实现类:

    • StandardMultipartHttpServletRequest//基于Servlet 3.0,可直接使用
    • CommonsMultipartResolver//依赖 Apache Commons FileUpload,需导包才能用

  2. LocaleResolver(context);

    处理国际化
public interface LocaleResolver {
	//根据request对象根据指定的方式获取一个Locale，如果没有获取到，则使用用户指定的默认的Locale
	Locale resolveLocale(HttpServletRequest request);
	//用于实现Locale的切换。比如SessionLocaleResolver获取Locale的方式是从session中读取，但如果
	//户想要切换其展示的样式(由英文切换为中文)，那么这里的setLocale()方法就提供了这样一种可能
	void setLocale(HttpServletRequest request, @Nullable HttpServletResponse response, @Nullable Locale locale);
}

  3. ThemeResolver

  4. HandlerMapping

    根据request找到一个对应的Handler,返回HandlerExecutionChain

    作用是根据当前请求的找到对应的 Handler，并将 Handler（执行程序）与一堆 HandlerInterceptor（拦截器,也是他来处理的）封装到 HandlerExecutionChain 对象中。返回给中央调度器.

  5. HandlerAdapter

    存在多个.

    mvc实际上是找到HandlerAdapter,用HandlerAdapter来执行对应的handler

    因为SpringMVC中的Handler可以是任意的形式，只要能处理请求就ok，但是Servlet需要的处理方法的结构却是固定的，都是以request和response为参数的方法。如何让固定的Servlet处理方法调用灵活的Handler来进行处理呢？这就是HandlerAdapter要做的事情。

    public interface HandlerAdapter {
	//当前 HandlerAdapter 是否支持这个 Handler
	boolean supports(Object handler);
	//调用handle处理这个请求，然后返回ModelAndView 对象
	@Nullable
	ModelAndView handle(HttpServletRequest request, HttpServletResponse response, Object handler) throws Exception;
	long getLastModified(HttpServletRequest request, Object handler);
}

  6. HandlerExceptionResolver

    用于处理Mvc过程中出现的异常,此组件可以返回异常对应的MAV,给render渲染出页面.

    官方推荐使用@ExceptionHandler

  7. RequestToViewNameTranslator

    从request中获取viewName.

    只有一个默认实现:DefaultRequestToViewNameTranslator

  8. ViewResolver

    ViewResolver用来将String类型的视图名和Locale解析为View类型的视图。View是用来渲染页面的，也就是将程序返回的参数填入模板里，生成html（也可能是其它类型）文件。这里就有两个关键问题：使用哪个模板？用什么技术（规则）填入参数？这其实是ViewResolver主要要做的工作

    ViewResolver需要找到渲染所用的模板和所用的技术（也就是视图的类型）进行渲染，具体的渲染过程则交由不同的视图自己完成。

  9. FlashMapManager

    用来管理FlashMap的，FlashMap主要用在redirect中传递参数。
