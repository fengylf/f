spring之循环依赖

循环依赖需要知道的几点:

  • 必须达成的共识, spring 的单例池中只能放 bean,而走完实例化-属性填充-初始化三个阶段后(后文统一称作生命周期流程,当然我们忽略了销毁这个阶段)的对象才叫 bean,才能放入单例池,半成品不能放到单例池,很重要的共识,否则很多问题没法讨论.
  • 对用户来说,A,B 互相做为对方的属性,他们两个发生了循环依赖
  • 对 spring 来说,A,B 互相做为对方的属性,先调用了 getBean(“a”),那么 A 才真正的出现了循环依赖.对 spring 的流程来说,B 并没有发生循环依赖.因为 B 在生命周期过程中,会发现 A 已经存在,可以直接注入(虽然是早期对象,三级工厂缓存),这一点比较重要.

DefaultSingletonBeanRegistry

三级缓存介绍

ConcurrentHashMap<String, Object> singletonObjects : 一级缓存,spring 容器 ,单例池,存的是 spring 的 bean,
HashMap<String, ObjectFactory<?>> singletonFactories:提前暴露一个工厂(如果直接提前暴露对象,后面没办法完成 aop),主要为了解决循环依赖的 aop. 3 级缓存,工厂,过程比较复杂,如果同一个 bean 被多个其它的 bean 引用,只用二级缓存的话性能低.引入三级缓存.
HashMap<String, Object> earlySingletonObjects: 2 级缓存,存的是对象,或者理解为早期的非完整 bean,完整之后会放到一级缓存

为什么

为什么单例池 singletonObjects 是 ConcurrentHashMap,而另外两个是 HashMap?

  • singletonObjects 是允许你多线程的访问的;而且当能取到值的时候,它是不加锁的.当取到为 null 的时候会自动替你上锁同步.
  • 另外两个缓存你要读取的前提是当前 bean 正在创建,而且读取的时候是加锁同步的,也就是本身就限制了只有一个线程可以访问到其他两个缓存.
  • 新版 spring 把 earlySingletonObjects 也改成了 ConcurrentHashMap,把单例池和二级缓存都进行了double checked locking.

三级缓存的秘密:

前提:spring 的基本原则是,走完生命周期流程,才叫 bean

  • 首先 singletonObject 单例池肯定要只能放走完流程的 bean,保证没走完流程的只能获取到 null,(而且保证获取到 null 后,想走 getSingleton()创建,还得加锁).因此半成品不能放单例池,所有至少要有另外一个缓存来存放不完整的对象.
  • 两个缓存可以解决不牵涉 aop 的普通 bean 的循环依赖.
  • 第三个缓存只是用来解决 aop 的 bean 之间的循环依赖.
  • 我的理解,引入第三个缓存的意义就是 aop 过程中的一个标记作用:earlySingletonObjects 有值就标记了 singletonFactories 被访问过一次,如果有 aop 的话,他一定被增强过了
  • 其次,isSingletonCurrentlyInCreation()是要保证,只有循环依赖的情况才能进入,那么单独一个二级缓存肯定不行;然而你不能从二级缓存直接取出放到单例池,因为此时它没走完流程. 否则 A 第二次进入 getSingleton()之后,要把二级缓存放到单例池,但明显此时还是半成品,还是不能放到单例池.
  • 因此必须要三级缓存.保证 A 第二次进来的时候,将半成品放到二级缓存,保证其他线程来还是不能从单例池取出 A.
  • spring 的三级缓存实现了这个效果:解决了 aop 的循环依赖,通过提前 aop 的发生.是在完整流程最后一个阶段发生的 aop,即在 postProcessAfterInitialization()方法中完成的.不影响前面的流程.
  • 第三级缓存是不是必须的呢,原则上不是必须的,但它不是多余的,你可以换个方式来实现,比如你可以为第二级缓存中的对象做一个标记,标志它是否已经发生过 aop,保证它只会被增强一次(例如 A 和 B 循环依赖,但 B 又同时依赖 C,C 又依赖 A,如果你不加标志位,A 就会被增强两次),而不会发生重复增强,那么就不需要第三级缓存,但是这样就比较绕.

AOP 时的循环依赖:A&B

代理类$A,$B

  • A 先实例化,包装成工厂之后加入到三级缓存 singletonFactories.
  • A 开始属性注入,发现依赖 B
  • 去实例化 B,然后包装成工厂之后加入到三级缓存 singletonFactories.
  • B 开始属性注入,发现依赖 A
  • 又去 getBean(“a”),进而 getSingleton(“a”, true),然后发现三级缓存中有 A,就调用 singletonFactory.getObject(),也就是调用 getEarlyBeanReference(“a”, mbd, bean)
  • 回调SmartInstantiationAwareBeanPostProcessor(其中_AbstractAutoProxyCreator).getEarlyBeanReference(A@001, “a”),将 A 存入缓存中,_this.earlyProxyReferences.put(cacheKey, bean);
  • 发现需要增强 A,增强后得到$A@15ca7889 返回,并放入二级缓存 earlySingletonObjects
  • 从而顺利为 B 注入代理后的$A@15ca7889.
  • B 属性注入完成之后,开始 initializeBean(),然后又调到了 AbstractAutoProxyCreator#postProcessAfterInitialization(),发现 B 也需要被增强,得到 exposedObject = $B@d21a74c
  • 通过 addSingleton(“b”, $B@d21a74c),将三级缓存中的B移除,并$B@d21a74c 放入单例池
  • 至此 B 的 bean 正式完成——————————-
  • 回到 A 的属性注入流程,此时 exposedObject 还是原始是 A@001,将$B@d21a74c 注入属性后,进入 initializeBean()
  • 然后调到 AbstractAutoProxyCreator#postProcessAfterInitialization(),由于this.earlyProxyReferences 已经缓存了上面的结果,此时不必再次增强了,原样返回即可(因为可以从二级缓存中取)
  • exposedObject = 还是 A@001
  • getSingleton(“a”, false),去取出二级缓存中的$A@15ca7889,
  • exposedObject = earlySingletonReference,即$A@15ca7889,返回出去;

总结:

  • AbstractAutoProxyCreator 通过两个方法来完成 aop 间的循环依赖.postProcessAfterInitialization()和 getEarlyBeanReference()
  • A 是第二次进入 getBean()后,通过后置处理器增强的
  • B 是 initializeBean()阶段增强的
  • 最后要注意的是,代理后的对象内的属性是 null,而内部的原始父类引用中的属性才有值,因为是属性注入后才 aop,而 aop 之后则没有赋值操作了.

自己对三级缓存的理解:

  • 不要缓存?
    • 前提是错误的抛弃掉共识,把未经完整流程的对象放入单例池做为 bean
    • 什么都不考虑的话,不考虑 aop,不用缓存就可以解决循环依赖(前提是单线程,否则可能拿到未初始化完成的 bean),这么做不太合适
    • 如果要 aop,则不行.(??或者实例化之后,直接提前 aop,然后放入单例池.然后对 aop 后的代理对象进行属性填充,初始化,这样不太好?例如你的初始化方法早早的就被代理了,如日志,那么你连初始化方法都会提前增强,想法有点奇葩)
  • 二级缓存,不考虑 aop?
    • 将早期对象 A 的工厂,放入二级缓存,然后对原始对象属性填充,如果没有发生循环依赖,完成流程,直接加入单例池中.
    • 待发生循环依赖,第二次 getBean()将从二级缓存中取出早期原始对象,赋值给 B 的属性.
    • 这样就可以保证正常没有 aop 的循环依赖,保证单例池中都是完整流程的 bean
  • 二级缓存,加上 aop?
    • 将早期对象 A 的工厂,放入二级缓存

进入 doGetBean()方法的时候,会先检查缓存,再决定是否需要 create 流程.能拿到自然就不走了.
某个 bean 创建过程,doCreateBean()方法的时候,已创建出 beanWrapper,还没有 populateBean()之前,就将早期对象包装成工厂,放入三级缓存 singletonFactories(三级缓存放的是工厂)中.

getSingleton(String beanName, boolean allowEarlyReference)

//DefaultSingletonBeanRegistry#
protected Object getSingleton(String beanName, boolean allowEarlyReference) {
		// 先检查单例池,单例池存在就说明走过完整 bean 生命周期直接返回了.
		Object singletonObject = this.singletonObjects.get(beanName);
		if (singletonObject == null && isSingletonCurrentlyInCreation(beanName)) {
            //如果从单例缓存中获取不成功，则锁定全局变量,这里只会在循环依赖时进入
            //检查二级缓存,早期对象
			singletonObject = this.earlySingletonObjects.get(beanName);
            //双重检查锁（double checked locking）
			if (singletonObject == null && allowEarlyReference) {
				synchronized (this.singletonObjects) {
					// Consistent creation of early reference within full singleton lock
					singletonObject = this.singletonObjects.get(beanName);
					if (singletonObject == null) {
						singletonObject = this.earlySingletonObjects.get(beanName);
						if (singletonObject == null) {
                            //前两级缓存都没有,从三级缓存找
							ObjectFactory<?> singletonFactory = this.singletonFactories.get(beanName);
							if (singletonFactory != null) {
                                //三级缓存拿到,就从工厂中取出早期对象,移动到二级缓存中
                                //只有循环依赖,第二次调用getBean()的那个bean
                               //(例如A.B循环依赖,先getBean(A),那么第二次getBean(A)才会走到这里)
								singletonObject = singletonFactory.getObject();
								this.earlySingletonObjects.put(beanName, singletonObject);
								this.singletonFactories.remove(beanName);
							}
						}
					}
				}
			}
		}
		return singletonObject;
	}

getSingleton(String beanName, ObjectFactory<?> singletonFactory)

public Object getSingleton(String beanName, ObjectFactory<?> singletonFactory) {
		Assert.notNull(beanName, "Bean name must not be null");
		synchronized (this.singletonObjects) {
			Object singletonObject = this.singletonObjects.get(beanName);
			if (singletonObject == null) {
				if (this.singletonsCurrentlyInDestruction) {
					throw new BeanCreationNotAllowedException(beanName,
							"Singleton bean creation not allowed while singletons of this factory are in destruction " +
							"(Do not request a bean from a BeanFactory in a destroy method implementation!)");
				}
				if (logger.isDebugEnabled()) {
					logger.debug("Creating shared instance of singleton bean '" + beanName + "'");
				}
				beforeSingletonCreation(beanName);
				boolean newSingleton = false;
				boolean recordSuppressedExceptions = (this.suppressedExceptions == null);
				if (recordSuppressedExceptions) {
					this.suppressedExceptions = new LinkedHashSet<>();
				}
				try {
					singletonObject = singletonFactory.getObject();
					newSingleton = true;
				}
				if (newSingleton) {
					addSingleton(beanName, singletonObject);
				}
			}
			return singletonObject;
		}
	}

aop 发生循环依赖的情况

AbstractAutoProxyCreator

earlyProxyReferences:只会存放第二次调用 getBean()的那个早期 bean 引用,即从三级缓存工厂中取出的时候,放入此 map;老版本 spring 用的 set 集合.

 // AbstractAutoProxyCreator#
@Override
//initialize最后调用,正常实现aop
	public Object postProcessAfterInitialization(@Nullable Object bean, String beanName) {
		if (bean != null) {
			Object cacheKey = getCacheKey(bean.getClass(), beanName);
            //第二次调用getBean()的那个早期bean引用,即从三级缓存工厂中取出的那个bean,
            //此处才会相等,就会跳过增强的检查.因为在getEarlyBeanReference()已经做过增强检查.
            //其他时候一般remove都是null
			if (this.earlyProxyReferences.remove(cacheKey) != bean) {
				return wrapIfNecessary(bean, beanName, cacheKey);
			}
		}
		return bean;
	}

	@Override
//提供循环依赖时,提前实现aop的功能
//在循环依赖时,在第二次调用getBean之后才会调用此方法,从三级缓得到早期bean引用
	public Object getEarlyBeanReference(Object bean, String beanName) {
		Object cacheKey = getCacheKey(bean.getClass(), beanName);
        //将bean的早期对象加到this.earlyProxyReferences,
		this.earlyProxyReferences.put(cacheKey, bean);
        //aop的时候会在此提前增强
		return wrapIfNecessary(bean, beanName, cacheKey);
	}
